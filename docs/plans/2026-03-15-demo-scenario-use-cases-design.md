## 1. 概述

本文档定义了CKY.MAF框架两个Demo项目（SmartHome智能家居和CustomerService智能客服）的会话场景用例设计。

### 1.1 设计背景

CKY.MAF是一个企业级多Agent框架，基于Microsoft Agent Framework构建。为了全面展示框架能力，我们需要设计一套完整的会话场景用例，用于：

- **产品演示**：向客户、投资者、合作伙伴展示框架核心能力
- **功能测试**：作为集成测试和E2E测试的基础
- **开发指导**：为开发团队提供清晰的实施目标
- **用户文档**：作为最终用户的使用手册和示例

### 1.2 设计原则

1. **真实导向**：所有用例基于真实用户旅程，贴近实际使用场景
2. **全面覆盖**：覆盖单Agent、多Agent协作、多轮对话、异常处理四个维度
3. **分层设计**：按用户旅程→用例实例→元数据标注三层结构组织
4. **易于查找**：通过多维度索引支持不同角色的查找需求

## 2. 设计目标

### 2.1 功能覆盖目标

| 维度 | SmartHome | CustomerService | 合计 |
|------|-----------|-----------------|------|
| 单Agent基础功能 | 12个 | 14个 | 26个 |
| 多Agent协作场景 | 14个 | 14个 | 28个 |
| 多轮对话管理 | 6个 | 8个 | 14个 |
| 异常处理和容错 | 4个 | 2个 | 6个 |
| **总计** | **36个** | **38个** | **74个** |

### 2.2 质量目标

- **演示价值**：至少70%的用例具有高演示价值（4星以上）
- **测试覆盖**：P0优先级用例（核心功能）≥ 30%
- **文档完整性**：所有用例包含完整的执行流程和预期结果
- **响应时间**：90%的用例响应时间 < 3秒

## 3. 整体架构

### 3.1 三层结构

```
┌─────────────────────────────────────────────┐
│         用户旅程层 (顶层组织)                │
│   8个核心旅程 × 2个Demo = 16个旅程          │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│         用例实例层 (中层组织)                │
│   4-6个用例/旅程 × 16个旅程 = 74个用例      │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│         元数据标注层 (底层支持)              │
│   6个维度索引：Agent、复杂度、优先级等      │
└─────────────────────────────────────────────┘
```

### 3.2 组织方式

**按用户旅程分类（主要组织方式）**：
- 优点：贴近真实使用，自然覆盖多Agent协作和多轮对话
- 适合：演示脚本、用户手册、端到端测试

**多维索引（辅助查找方式）**：
- 按Agent查找：开发团队快速定位相关用例
- 按复杂度查找：循序渐进的学习路径
- 按优先级查找：测试团队的测试计划
- 按演示价值查找：产品团队的演示准备

## 7. 元数据标注方案

### 7.1 YAML格式元数据

每个用例文件头部包含YAML格式的元数据：

```yaml
---
metadata:
  case_id: SH-MORNING-001
  journey: 早晨唤醒
  journey_order: 1

  # 分类标签
  case_type: primary  # primary | variant | exception | boundary
  domain: smarthome
  complexity: L2  # L1单Agent | L2多轮对话 | L3多Agent协作 | L4复杂编排

  # 覆盖范围

    - WeatherAgent

    - LightingAgent
  capabilities:
    - weather-query

  coverage:
    - single-agent: true
    - multi-agent: true
    - multi-turn: true
    - error-handling: false

  # 用途标注
  demo_value: 5  # 1-5星，演示价值
  test_priority: P0  # P0必须 | P1重要 | P2一般
  doc_importance: high  # high | medium | low

  # 执行属性
  estimated_duration_seconds: 45
  requires_external_service: true  # 是否依赖外部API
  requires_hardware: false  # 是否依赖真实硬件

  # 依赖关系
  depends_on: []
  enables: [SH-MORNING-002, SH-MORNING-003]

  # 状态
  status: designed  # designed | implementing | testing | completed
  assigned_to: ""
  last_updated: "2026-03-15"
---
```

### 7.2 元数据字段说明

#### 基础信息
- **case_id**: 用例唯一标识
- **journey**: 所属用户旅程
- **journey_order**: 在旅程中的顺序

#### 分类标签
- **case_type**: 用例类型
  - `primary`: 主流程，标准成功路径
  - `variant`: 变体流程，不同参数或选择
  - `exception`: 异常流程，错误处理
  - `boundary`: 边界测试，极端情况

- **complexity**: 复杂度级别
  - `L1`: 单Agent基础功能
  - `L2`: 多轮对话管理
  - `L3`: 多Agent协作
  - `L4`: 复杂编排和业务逻辑

#### 覆盖范围
- **agents**: 涉及的Agent列表
- **capabilities**: 涉及的能力标签
- **coverage**: 覆盖维度标注

#### 用途标注
- **demo_value**: 演示价值（1-5星）
- **test_priority**: 测试优先级（P0/P1/P2）
- **doc_importance**: 文档重要性（high/medium/low）

#### 执行属性
- **estimated_duration_seconds**: 预估执行时长
- **requires_external_service**: 是否依赖外部API
- **requires_hardware**: 是否依赖真实硬件

#### 依赖关系
- **depends_on**: 前置依赖的用例
- **enables**: 启用的后续用例

#### 状态管理
- **status**: 当前状态
- **assigned_to**: 负责人
- **last_updated**: 最后更新时间

## 9. 实施建议

### 9.1 分阶段实施计划

建议分**3个阶段**实施：

#### 第一阶段：核心用例（4周）

**目标**：实现所有P0优先级用例，确保核心功能可用。

**SmartHome** (12个)：
- 早晨唤醒（SH-MORNING-001, 004, 005）
- 离家准备（SH-DEPART-001, 003, 004）
- 回家放松（SH-HOME-001, 005）
- 紧急情况（SH-EMERG-001, 002, 003, 004, 005）

**CustomerService** (12个)：
- 初次咨询（CS-INITIAL-001, 004, 005）
- 订单查询（CS-ORDER-001, 002, 004）
- 退换货处理（CS-RETURN-001, 004）
- 投诉建议（CS-COMPLAIN-004）
- 工单跟进（CS-TICKET-001）
- 问题升级（CS-ESCAL-001, 003）

#### 第二阶段：扩展用例（4周）

**目标**：实现P1优先级用例，覆盖更多场景和变体。

包括所有P1标记的用例（约30个）。

#### 第三阶段：完善用例（2周）

**目标**：实现P2优先级用例，补充边界情况和优化。

包括所有P2标记的用例（约10个）。

### 9.2 实施注意事项

1. **用例文档先行**：
   - 在实现每个用例前，先编写完整的用例文档
   - 文档经过审核后再开始实现

2. **自动化测试覆盖**：
   - P0用例必须有完整的自动化测试
   - P1用例至少有主要路径的测试
   - P2用例可以手动测试

3. **演示脚本准备**：
   - 高演示价值用例（4星以上）准备演示脚本
   - 录制演示视频供团队学习

4. **持续迭代优化**：
   - 根据演示和测试反馈持续优化用例
   - 定期review用例的演示价值和优先级

5. **跨角色协作**：
   - 开发团队负责实现
   - 测试团队负责验证
   - 产品团队负责演示价值评估
   - 文档团队负责用户手册编写

### 9.3 成功标准

**第一阶段完成后**：
- ✅ 所有P0用例实现并通过测试
- ✅ 可以进行基本的框架演示
- ✅ 核心功能稳定可用

**第二阶段完成后**：
- ✅ 所有P0+P1用例实现
- ✅ 可以进行完整的客户演示
- ✅ 测试覆盖率达到80%

**第三阶段完成后**：
- ✅ 所有74个用例全部实现
- ✅ 测试覆盖率达到90%
- ✅ 完整的用户手册和演示材料
